# Архитектура VibeBuild

## Обзор системы

VibeBuild — это расширение для Koji, которое автоматизирует разрешение зависимостей при сборке RPM-пакетов. Система состоит из двух основных частей:

1. **VibeBuild CLI** — Python-приложение для анализа зависимостей и оркестрации сборки
2. **Инфраструктура Koji** — сервер сборки RPM-пакетов

## Компоненты VibeBuild

### 1. Анализатор (`analyzer.py`)

**Ответственность:** Парсинг SRPM и spec-файлов, извлечение метаданных.

```
┌─────────────────────────────────────────┐
│              Analyzer                    │
├─────────────────────────────────────────┤
│ + get_build_requires(srpm) -> list[str] │
│ + get_package_info_from_srpm() -> Info  │
├─────────────────────────────────────────┤
│              SpecAnalyzer               │
├─────────────────────────────────────────┤
│ + analyze_spec(path) -> PackageInfo     │
│ - _parse_spec_content(content)          │
│ - _parse_build_requires(line)           │
│ - _expand_macros(value)                 │
└─────────────────────────────────────────┘
```

**Поток данных:**
```
SRPM-файл ──► rpm2cpio ──► .spec-файл ──► SpecAnalyzer ──► PackageInfo
                                                              │
                                                              ▼
                                                    ┌─────────────────┐
                                                    │ - name          │
                                                    │ - version       │
                                                    │ - release       │
                                                    │ - build_requires│
                                                    │ - source_urls   │
                                                    └─────────────────┘
```

### 2. Резолвер имён (`name_resolver.py` + `ml_resolver.py`)

**Ответственность:** Разрешение виртуальных имён RPM-зависимостей в реальные имена пакетов.

Spec-файлы часто содержат имена зависимостей, не совпадающие с реальными именами RPM-пакетов:
- `python3dist(requests)` — виртуальный provide, реальный пакет: `python3-requests`
- `%{python3_pkgversion}-devel` — нераскрытый макрос, реальный пакет: `python3-devel`
- `pkgconfig(glib-2.0)` — pkgconfig provide, реальный пакет: `glib-2.0-devel`

Резолвер имён обрабатывает это через многофазный конвейер:

```
┌─────────────────────────────────────────────────────────────────┐
│                   PackageNameResolver                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│   Вход ──► Кэш ──► Раскрытие макросов ──► Паттерны provides    │
│               │           │                   │                  │
│           (попад.)   SYSTEM_MACROS      PROVIDE_PATTERNS        │
│               │       (18 макросов)    (9 regex-паттернов)      │
│               ▼           │                   │                  │
│           Результат  ──►──┤──────►─────►──────┤                  │
│                           │                   │                  │
│                           ▼                   ▼                  │
│                     ┌──────────────────────────────┐             │
│                     │  ML-резерв (опционально)     │             │
│                     │  TF-IDF + KNN (scikit-learn) │             │
│                     └──────────────────────────────┘             │
│                           │                                      │
│                           ▼                                      │
│                     Разрешённое имя                               │
└─────────────────────────────────────────────────────────────────┘
```

**Компоненты:**

- `PackageNameResolver` — резолвер на основе правил с раскрытием макросов, паттернами виртуальных provides, маппингом имён SRPM и ML-резервом
- `MLPackageResolver` — опциональная ML-модель, использующая TF-IDF символьные n-граммы (2-5) и K-Nearest Neighbors с косинусным расстоянием. Обучена на маппингах provides-to-package из Fedora

**Точки интеграции:**

- `Analyzer` использует `SYSTEM_MACROS` для лучшего раскрытия макросов
- `Resolver` нормализует имена перед проверкой в Koji
- `Fetcher` использует `resolve_srpm_name()` для перебора вариантов имён SRPM
- `Builder` создаёт и передаёт резолвер во все компоненты

**ML-модель опциональна:** Если scikit-learn не установлен, ML-резерв тихо деградирует. Установка: `pip install vibebuild[ml]`.

### 3. Резолвер зависимостей (`resolver.py`)

**Ответственность:** Проверка доступности зависимостей в Koji, построение графа зависимостей.

```
┌───────────────────────────────────────────────────┐
│            DependencyResolver                      │
├───────────────────────────────────────────────────┤
│ + __init__(koji_client, koji_tag, name_resolver)  │
│ + find_missing_deps(deps, tag) -> list            │
│ + build_dependency_graph(pkg, srpm)               │
│ + topological_sort() -> list[str]                 │
│ + get_build_chain() -> list[list[str]]            │
├───────────────────────────────────────────────────┤
│              KojiClient                            │
├───────────────────────────────────────────────────┤
│ + list_packages(tag) -> list[str]                 │
│ + list_tagged_builds(tag) -> dict                 │
│ + package_exists(pkg, tag) -> bool                │
└───────────────────────────────────────────────────┘
```

**Интеграция с разрешением имён:** `find_missing_deps()` нормализует имена зависимостей через `name_resolver.resolve()` перед проверкой в Koji. При неудаче используется исходное имя.

**Алгоритм построения DAG:**

```
1. Начинаем с корневого пакета
2. Для каждого пакета:
   a. Если пакет существует в теге Koji → помечаем как доступный
   b. Иначе:
      - Извлекаем BuildRequires
      - Находим недостающие зависимости
      - Рекурсивно обрабатываем каждую зависимость
3. Строим граф зависимостей
4. Выполняем топологическую сортировку
```

**Структура данных:**

```
DependencyNode:
  - name: str
  - srpm_path: Optional[str]
  - dependencies: list[str]
  - is_available: bool
  - build_order: int

Пример графа:
                    ┌─────────┐
                    │ my-app  │
                    └────┬────┘
                         │
           ┌─────────────┼─────────────┐
           ▼             ▼             ▼
      ┌─────────┐   ┌─────────┐   ┌─────────┐
      │ lib-foo │   │ lib-bar │   │ lib-baz │
      └────┬────┘   └─────────┘   └────┬────┘
           │        (доступен)         │
           ▼                           ▼
      ┌─────────┐                 ┌─────────┐
      │lib-base │                 │lib-core │
      └─────────┘                 └─────────┘
       (доступен)                 (доступен)

Порядок сборки: [lib-foo, lib-baz, my-app]
Цепочка сборки: [[lib-foo, lib-baz], [my-app]]
```

### 4. Загрузчик (`fetcher.py`)

**Ответственность:** Скачивание SRPM из внешних источников.

```
┌───────────────────────────────────────────────────────────┐
│              SRPMFetcher                                   │
├───────────────────────────────────────────────────────────┤
│ + __init__(download_dir, sources, ..., name_resolver)     │
│ + download_srpm(name, version) -> path                    │
│ + search_fedora_src(name) -> list                         │
│ + get_package_versions(name) -> list                      │
├───────────────────────────────────────────────────────────┤
│ - _download_from_koji(...)                                │
│ - _download_from_src(...)                                 │
│ - _extract_sources(spec)                                  │
└───────────────────────────────────────────────────────────┘
```

**Разрешение имён SRPM:** Когда предоставлен `name_resolver`, `download_srpm()` использует `resolve_srpm_name()` для генерации нескольких вариантов имён SRPM. Например, `python3-requests` пробуется и как `python-requests`, и как `python3-requests`. Каждый вариант пробуется по всем источникам перед переходом к следующему.

**Источники SRPM (в порядке приоритета):**

1. **Fedora Koji** (`koji.fedoraproject.org`)
   - Метод: `koji download-build --type=src`
   - Преимущество: готовые SRPM

2. **src.fedoraproject.org**
   - Метод: скачивание spec + исходников, локальная сборка SRPM
   - Используется как запасной вариант

### 5. Сборщик (`builder.py`)

**Ответственность:** Оркестрация сборки в Koji.

```
┌─────────────────────────────────────────┐
│              KojiBuilder                │
├─────────────────────────────────────────┤
│ + build_package(srpm, wait) -> Task     │
│ + build_with_deps(srpm) -> BuildResult  │
│ + wait_for_repo(tag, timeout) -> bool   │
│ + build_chain(packages) -> BuildResult  │
├─────────────────────────────────────────┤
│ - _run_koji(*args) -> CompletedProcess  │
│ + get_build_status(task_id) -> Status   │
│ + cancel_build(task_id) -> bool         │
└─────────────────────────────────────────┘
```

**Алгоритм build_with_deps:**

```
┌──────────────────────┐
│ 1. Парсинг SRPM      │
└──────────┬───────────┘
           │
           ▼
┌──────────────────────┐
│ 2. Построение графа  │
│    зависимостей      │
└──────────┬───────────┘
           │
           ▼
┌──────────────────────┐
│ 3. Получение цепочки │
│    сборки            │
└──────────┬───────────┘
           │
           ▼
┌──────────────────────────────────────────┐
│ 4. Для каждого уровня цепочки:          │
│    a. Собрать все пакеты уровня         │
│    b. Дождаться завершения сборок       │
│    c. Дождаться регенерации репозитория  │
└──────────────────────┬───────────────────┘
           │
           ▼
┌──────────────────────┐
│ 5. Собрать целевой   │
│    пакет             │
└──────────────────────┘
```

### 6. CLI (`cli.py`)

**Ответственность:** Пользовательский интерфейс командной строки.

```
Команды:
  vibebuild TARGET SRPM                   # Сборка с зависимостями
  vibebuild --analyze-only SRPM           # Только анализ
  vibebuild --download-only PKG           # Только скачивание
  vibebuild --dry-run TARGET SRPM         # Показать план

Опции разрешения имён:
  --no-name-resolution                    # Отключить нормализацию имён
  --no-ml                                 # Отключить ML (только правила)
  --ml-model PATH                         # Пользовательский файл модели
```

## Обработка ошибок

### Логика повторных попыток

```python
RETRY_CONFIG = {
    "koji_build": {
        "max_retries": 3,
        "backoff": "exponential",
        "initial_delay": 10,
    },
    "download_srpm": {
        "max_retries": 2,
        "backoff": "linear",
        "initial_delay": 5,
    },
}
```

## Производительность

### Кэширование

- **Кэш доступных пакетов:** список пакетов в теге Koji кэшируется
- **Кэш скачанных SRPM:** скачанные SRPM сохраняются для повторного использования
- **Кэш графа зависимостей:** граф зависимостей строится один раз
- **Кэш разрешения имён:** разрешённые имена кэшируются в памяти (на сессию)
- **Кэш ML-предсказаний:** ML-предсказания кэшируются в `~/.cache/vibebuild/ml_name_cache.json` (персистентный между сессиями)

### Параллелизм

- Пакеты на одном уровне графа зависимостей могут собираться параллельно
- Скачивание SRPM может выполняться параллельно с анализом

## Безопасность

### Аутентификация

- SSL клиентские сертификаты для Koji
- Проверка CA для HTTPS-соединений

### Изоляция сборки

- Все сборки выполняются в mock chroot
- Изолированная сетевая среда на builder
